MS Project import conflict resolution rules

When you import an MS Project plan there can be conflicts with existing tasks. Track+ follows specific rules to resolve such conflicts.

During synchronization of an MS Project file with a Track+ database, conflicts can occur. This section describes the main conflict handling scenarios.

If a task exists in Track+, and has been deleted from MS Project, you are asked if you want to mark this task as deleted in Track+ as well, as long as it is still open.

If a task has been deleted in Track+, but still exists in MS Project, you will be asked if you want to restore the task in Track+

The following fields can be modified in either Track+ or MS Project, and the changes will be synchronized:

  • Planned value for work

  • Remaining estimated work

  • Start date

  • End date (due date)

  • Task dependency links

  • Task title

When either of work, start date or end date was modified in Track+ after the last MS Project modification date, you will be able to choose the correct value during the import conflict resolution process.

Conflict resolution for task dependency links is solely based on the last modification date. If a dependency link was modified in MS Project, it overwrites the Track+ link if the MS Project file was modified later than the link was modified in Track+. Otherwise the Track+ link will be left unchanged.

The same behaviour holds if links were deleted. If a link was deleted from the MS Project file it will also be deleted from Track+ if the MS Project file modification came later than the last modification if the Track+ link.

When there is a conflict with the tasks title, the Track+ title is left unchanged.

Data for actual work is never imported from MS Project files into Track+, since this is data that is typically modified by a large number of team members, and not by a project manager. Actual work is however exported to MS Project files so that it can be analyzed by the project administrator.
